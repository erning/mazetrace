# MazeTrace 设计文档

## 1. 项目概述

`MazeTrace` 是一个运行在 Terminal 中的迷宫生成与探索可视化应用。程序启动后先从一张四面封闭的网格开始，用动画展示算法逐步“拆墙”生成迷宫的过程。迷宫生成完成后，入口位于左上角，出口位于右下角，程序默认等待用户按 `Space`，再用动画展示算法如何从入口尝试探索到出口。

项目名称确定为 `MazeTrace`。

## 2. 设计目标

- 在 Terminal 中提供完整的迷宫生成、展示和探索体验。
- 使用 Unicode 线框字符绘制迷宫墙体，让迷宫看起来像一张清晰的图纸。
- 动态展示迷宫生成过程，包括选择单元格、拆墙、前进和回溯。
- 动态展示探索过程，包括前进、遇到死路、回溯和最终路径。
- 支持基础交互，例如暂停、单步、调速、重新生成和退出。
- 第一版使用 Rust + ratatui 实现，依赖保持克制，便于实现和维护。

## 3. 非目标

- 第一版不做图形界面。
- 第一版不做在线排行榜、存档、关卡编辑器或多人功能。
- 第一版不追求复杂的游戏机制，重点是“生成迷宫”和“观察探索过程”。
- 第一版不要求支持所有 Terminal 环境，但应尽量在常见 macOS、Linux Terminal 中正常显示。

## 4. 项目名称

项目名称为 `MazeTrace`。

命名含义：

- `Maze` 表示迷宫。
- `Trace` 表示探索轨迹、搜索过程和最终路径。

这个名字适合命令行程序、README、包名和后续扩展。Rust crate 名称建议使用小写的 `mazetrace`。

## 5. 用户体验

程序启动后直接进入迷宫界面，不显示落地页或说明页。用户先看到一张尚未打通的网格，然后看到墙体被逐步拆除，迷宫一点点形成。生成完成后，界面进入 `Ready` 阶段，显示完整迷宫并等待用户按 `Space` 开始探索。若启动时传入 `--auto-start`，程序会保持连续演示，生成完成后直接进入探索阶段。

默认流程如下：

1. 程序创建一张所有墙体都存在的初始网格。
2. 生成器从左上角单元格开始，逐步选择邻居并拆除两格之间的墙。
3. 界面显示当前生成位置和刚刚被拆除的墙。
4. 生成器遇到死路时回溯，直到所有单元格都被访问。
5. 迷宫生成完成后，界面标出入口 `S` 和出口 `E`。
6. 程序等待用户按 `Space`，或者在 `--auto-start` 模式下直接开始。
7. 探索者从入口出发，逐步尝试移动。
8. 已探索区域显示为 `·`。
9. 当前探索位置显示为 `●` 或 `@`。
10. 遇到死路时，探索者回溯。
11. 找到出口后，高亮最终路径。
12. 用户可以重新生成迷宫、切换速度或退出。

ratatui 支持颜色和样式后，生成阶段可以用高亮样式标出“刚刚被拆掉的墙”。如果用户使用无颜色环境，仍应通过墙体消失和 `○` 的移动看出生成过程。

## 9. 迷宫模型

迷宫由单元格组成。每个单元格保存四面墙：

```text
Cell
- row
- col
- north_wall
- east_wall
- south_wall
- west_wall
```

逻辑尺寸使用“单元格数”，例如 `21x11` 表示宽 21 个单元格、高 11 个单元格，不表示实际渲染字符数。

入口和出口是墙上的开口：

- 入口：左上角单元格的西墙打开，位置为 `(0, 0)`。
- 出口：右下角单元格的东墙打开，位置为 `(height - 1, width - 1)`。

第一版建议生成“完美迷宫”，也就是任意两个单元格之间只有一条路径。这样探索过程容易理解，不会出现环路带来的复杂行为。

## 10. 迷宫生成算法

生成阶段应被设计成可单步推进的状态机，而不是一次性生成完整迷宫。这样用户才能看到“拆墙”或“立墙”的过程，也方便暂停、继续和单步执行。

### 拆墙模型

大多数生成算法都从一张“所有单元格四面有墙”的网格开始。每次拆墙都发生在两个
相邻单元格之间，程序先判断方向，再同时更新两边的共享墙。例如从 `(row, col)`
向东拆墙时，会关闭当前单元格的东墙，也会关闭右侧单元格的西墙。

这个操作由 `Maze::carve_between(from, to)` 表示。它只接受相邻单元格；如果
两个位置不相邻，就不会改变迷宫。这样可以保证数据结构始终一致，不会出现“一侧
已打通、另一侧仍有墙”的状态。

拆墙类算法的共同目标，是在不破坏连通性的前提下，把新的单元格接入迷宫。DFS、
Prim、Kruskal、Aldous-Broder 和 Wilson 都通过不同策略选择下一堵要拆的墙。
最终生成的第一版迷宫是“完美迷宫”：所有单元格连通，任意两个单元格之间只有一条
路径。

```text
从一个已访问区域开始
选择一对相邻单元格
如果这次连接符合算法约束：
    拆掉两个单元格之间的共享墙
    将新单元格或新集合并入迷宫
重复直到所有单元格连通
```

Recursive Division 是例外。它先打开全部内部墙，再通过
`Maze::add_wall_between(from, to)` 逐步加墙分割区域，并为每道分割墙保留一个
通道。

当前支持这些生成算法：

| 算法 | 参数值 | 特点 |
| --- | --- | --- |
| DFS 回溯 | `dfs` | 长走廊较多，实现简单，动画像在迷宫里钻洞。 |
| Prim | `prim` | 从已生成区域向外扩张，分支更均匀。 |
| Kruskal | `kruskal` | 随机拆墙并合并集合，适合展示连通性建立过程。 |
| Aldous-Broder | `aldous-broder` | 随机游走直到覆盖所有单元格，理论均匀但动画会更慢、更飘。 |
| Wilson | `wilson` | 使用 loop-erased random walk，结果均匀，动画能展示随机走和擦环。 |
| Recursive Division | `recursive-division` | 从开放区域逐步加墙，视觉效果和拆墙算法明显不同。 |

Recursive Division 很适合展示，因为它像在空地上逐步“立墙”，和拆墙算法的观感差异明显。实现时先打开所有内部墙，再逐步加分割墙并为每道墙保留一个通道。

生成器状态包括：

```text
MazeGenerator
- current_cell
- stack
- visited_cells
- last_event
- status
- step_count
```

`last_event` 用于告诉渲染层上一帧发生了什么，例如：

```text
GenerationEvent
- Visit(cell)
- Carve(from_cell, to_cell, removed_wall)
- Wall(from_cell, to_cell, added_wall)
- Backtrack(from_cell, to_cell)
- Done
```

渲染层可以根据 `Carve` 事件短暂高亮刚被拆除的墙，或者高亮新打通的通道。高亮只属于视图状态，不应改变迷宫数据结构。

DFS 生成流程：

1. 从入口单元格开始。
2. 随机选择一个尚未访问的相邻单元格。
3. 打通当前单元格与目标单元格之间的墙。
4. 移动到目标单元格，并继续重复。
5. 如果当前单元格没有未访问邻居，则回溯到上一个单元格。
6. 直到所有单元格都被访问。

后续可加入其他生成算法，例如 Eller's Algorithm 或 Sidewinder，用于展示逐行生成的迷宫风格。

## 11. 探索算法

当前支持这些探索算法。生成算法和探索算法可以都叫 DFS，但用途不同：

- 生成 DFS：用于创建迷宫结构。
- 探索 DFS：用于从入口寻找出口。

| 算法 | 参数值 | 特点 |
| --- | --- | --- |
| DFS | `dfs` | 一条路走到底，遇到死路回溯，不保证最短路径。 |
| BFS | `bfs` | 像水波一样向外扩散，在等权迷宫中能找到最短路径。 |
| A* | `astar` | 使用曼哈顿距离启发式，更有目标感。 |
| Dijkstra | `dijkstra` | 按当前最短距离扩展，在等权迷宫中效果接近 BFS。 |
| Dead-End Filling | `dead-end` | 逐步填掉死路，最后留下入口到出口的路径。 |
| Wall-Follower | `wall-follower` | 模拟左手扶墙行走，在简单连通迷宫中能走到出口。 |

探索器状态包括：

```text
Explorer
- current_cell
- stack
- visited_cells
- parent_map
- final_path
- status
- step_count
```

DFS 探索流程：

1. 从入口单元格开始。
2. 标记当前单元格为已探索。
3. 检查可以到达且未访问的邻居。
4. 如果存在邻居，选择一个邻居前进，并记录父节点。
5. 如果不存在邻居，回溯到上一个单元格。
6. 如果到达出口，根据 `parent_map` 还原最终路径。
7. 将最终路径显示为双线字符，例如 `═`、`║`、`╔`、`╗`、`╚` 和 `╝`。

其他探索算法也应转换为可单步播放的事件序列，保持暂停、单步和最终路径渲染行为一致。

## 12. 渲染设计

渲染层负责把迷宫数据转换成 Terminal 字符画。

建议使用以下渲染比例：

```text
每个迷宫单元格：
- 内部高度：1 行
- 内部宽度：3 列
- 墙体占用边界行和边界列
```

如果迷宫尺寸为 `width x height`，渲染尺寸约为：

```text
render_width = width * 4 + 1
render_height = height * 2 + 1
```

墙体渲染步骤：

1. 创建一个二维字符画布。
2. 根据每个单元格的四面墙，在画布上画水平线和垂直线。
3. 根据每个连接点上下左右是否有墙，选择正确的 Unicode 连接字符。
4. 在入口和出口处移除对应外墙。
5. 在单元格内部绘制 `S`、`E`、`○`、`●`、`·`，并用双线字符连接最终路径。

连接点字符选择规则：

| 连接方向 | 字符 |
| --- | --- |
| 右、下 | `┌` |
| 左、下 | `┐` |
| 右、上 | `└` |
| 左、上 | `┘` |
| 上、下 | `│` |
| 左、右 | `─` |
| 上、下、右 | `├` |
| 上、下、左 | `┤` |
| 左、右、下 | `┬` |
| 左、右、上 | `┴` |
| 上、右、下、左 | `┼` |

如果某些 Terminal 字体对 Unicode 线框支持不佳，可以提供 ASCII 降级模式：

| Unicode | ASCII |
| --- | --- |
| `─` | `-` |
| `│` | `|` |
| 连接点 | `+` |
| `●` | `@` |
| 最终路径双线 | `-`、`|`、`+` |

## 13. 应用状态

应用可以维护一个全局状态对象：

```text
AppState
- maze
- generator
- explorer
- generation_algorithm
- exploration_algorithm
- speed
- paused
- show_final_path
- terminal_width
- terminal_height
- auto_sized
- render_fits
- message
```

主要状态值：

| 状态 | 含义 |
| --- | --- |
| `Generating` | 正在生成迷宫，并展示拆墙过程。 |
| `Ready` | 迷宫已生成，等待开始探索。 |
| `Exploring` | 正在探索。 |
| `Solved` | 已找到出口。 |
| `Failed` | 未找到出口，理论上完美迷宫不会出现。 |

暂停不作为独立状态值，而是由 `paused` 布尔标志表示。这样生成阶段、探索阶段和
已完成阶段都可以保留自己的业务状态，同时统一处理暂停、单步和恢复。

## 14. 屏幕尺寸策略

Terminal 尺寸不足时，程序不应该渲染错乱的迷宫。

推荐策略：

- 启动时检测 Terminal 宽高。
- 默认不使用固定迷宫尺寸，而是根据当前 Terminal 可用区域计算能完整显示的最大逻辑尺寸。
- 如果用户手动指定的尺寸过大，则显示提示信息。
- 状态栏始终保留至少 3 行。
- 迷宫区域超出屏幕时，第一版不做滚动，直接要求用户缩小迷宫或放大窗口。

默认尺寸计算：

```text
terminal_width = 当前 Terminal 列数
terminal_height = 当前 Terminal 行数

app_border_width = 2
app_border_height = 2
status_height = 3

maze_area_width = terminal_width - app_border_width
maze_area_height = terminal_height - app_border_height - status_height

width = floor((maze_area_width - 1) / 4)
height = floor((maze_area_height - 1) / 2)
```

其中 `width` 和 `height` 是迷宫逻辑尺寸，单位是单元格。计算结果应设置下限，例如最小不低于 `5x5`。如果 Terminal 小到无法容纳最小迷宫，程序显示提示信息，而不是尝试绘制错乱的界面。

如果用户只指定 `--width` 或 `--height` 其中一个值，另一个值仍然根据 Terminal 可用区域自动计算。如果用户同时指定两个值，则以用户指定值为准，并在尺寸过大时提示无法完整显示。

运行时窗口尺寸变化：

- 使用 `crossterm` 监听 `Event::Resize(width, height)`。
- 收到 resize 事件后更新 `terminal_width` 和 `terminal_height`，重新计算 ratatui 布局和迷宫渲染区域。
- 当前迷宫的逻辑尺寸不自动改变。无论处于生成阶段还是探索阶段，都应保持当前迷宫、生成器和探索器状态不变。
- 如果窗口变大，程序只扩大可用布局空间，并在状态栏提示“按 `N` 使用新尺寸重新生成迷宫”。是否生成更大的迷宫由用户决定。
- 如果窗口变小但仍能完整显示当前迷宫，程序直接按新布局重绘。
- 如果窗口变小后无法完整显示当前迷宫，程序自动暂停动画，显示居中的提示信息，要求用户放大窗口或按 `N` 重新生成适配当前窗口的小迷宫。
- 第一版不做迷宫滚动视口，也不裁剪迷宫内容。窗口太小时优先显示清晰提示，而不是显示残缺迷宫。

resize 处理原则：

- 自动尺寸计算只在创建新迷宫时生效，包括首次启动和用户按 `N` 重新生成。
- 运行中的 resize 只影响布局，不改变迷宫数据。
- `--width` 和 `--height` 指定的尺寸优先级高于自动尺寸，resize 后也不自动改写用户指定值。
- resize 事件可能连续触发，应做轻量处理：记录最新尺寸，在下一帧统一重绘即可。

## 15. 命令行参数

第一版可以支持这些参数。默认情况下不传 `--width` 和 `--height`，程序会根据 Terminal 最大可用区域自动计算迷宫逻辑尺寸；传入后才覆盖自动计算结果。

```text
--width 21
--height 11
--speed 60
--generator wilson
--solver dijkstra
--auto-start
--ascii
--seed 1234
```

参数说明：

| 参数 | 含义 |
| --- | --- |
| `--width` | 覆盖自动计算的迷宫宽度，单位是单元格。 |
| `--height` | 覆盖自动计算的迷宫高度，单位是单元格。 |
| `--speed` | 动画速度，数值越大越快。 |
| `--generator` | 生成算法，支持 `dfs`、`prim`、`kruskal`、`aldous-broder`、`wilson` 和 `recursive-division`。 |
| `--solver` | 探索算法，支持 `dfs`、`bfs`、`astar`、`dijkstra`、`dead-end` 和 `wall-follower`。 |
| `--auto-start` | 迷宫生成完成后直接开始探索，不停在 `Ready` 阶段。 |
| `--ascii` | 使用 ASCII 降级渲染。 |
| `--seed` | 指定随机种子，便于复现同一张迷宫。 |

旧版 `--algorithm` 仍作为 `--solver` 的隐藏兼容别名保留，命中时打印废弃提示。新文档
和帮助信息不再展示该参数。

## 16. 技术路线

第一版明确使用 Rust + ratatui 实现。

核心依赖建议：

| 依赖 | 用途 |
| --- | --- |
| `ratatui` | 绘制 Terminal UI，包括布局、边框、状态栏和迷宫区域。 |
| `crossterm` | 处理 Terminal 初始化、键盘事件、原始模式和退出恢复。 |
| `clap` | 解析命令行参数，可选但推荐。 |
| `rand` | 生成随机迷宫和支持随机种子。 |

技术约束：

- 使用 Cargo 管理项目。
- 程序入口为二进制命令 `mazetrace`。
- ratatui 只负责视图渲染，不承载迷宫生成和寻路逻辑。
- 通过 `crossterm` 处理键盘事件和 resize 事件。
- 迷宫生成、探索算法和渲染转换应保持可测试。
- 退出程序时必须恢复 Terminal 状态，避免留下原始模式或隐藏光标。

## 17. 模块划分

推荐结构：

```text
mazetrace/
- Cargo.toml
- src/
  - main.rs        # 程序入口、Terminal 初始化和错误处理
  - app.rs         # 应用状态和主循环
  - config.rs      # 命令行参数和默认配置
  - maze.rs        # 迷宫数据结构
  - generator.rs   # 迷宫生成算法和拆墙状态机
  - explorer.rs    # 探索算法
  - render.rs      # 迷宫到字符画布的转换
  - ui.rs          # ratatui 布局和组件绘制
```

键盘输入当前由 `app.rs` 的 `handle_key` 直接更新应用状态。等交互继续变复杂时，可以
再抽出轻量 `input.rs`，把 `KeyEvent` 转成应用动作枚举。

第一版也应保持基本模块拆分，避免把算法、输入和 ratatui 绘制都堆在 `main.rs` 中。

## 18. 主循环

主循环负责协调输入、探索和渲染。

每一帧执行：

1. 读取用户输入。
2. 根据输入更新 `AppState`。
3. 如果收到 `Event::Resize(width, height)`，更新 Terminal 尺寸、重新计算布局，并判断当前迷宫是否仍能完整显示。
4. 如果当前状态是 `Generating` 且没有暂停，执行一次或多次生成步骤。
5. 如果生成完成且没有启用 `--auto-start`，进入 `Ready` 阶段，等待用户按 `Space`。
6. 如果当前状态是 `Exploring` 且没有暂停，执行一次或多次探索步骤。
7. 使用 ratatui 重新绘制迷宫区域、状态栏和快捷键提示。
8. 按速度设置等待下一帧。

速度控制可以用“每秒步数”表示，例如：

```text
speed = 60 表示每秒最多执行 60 步
```

## 19. 实施里程碑

### 里程碑 1：生成动画

- 建立迷宫数据结构。
- 实现可单步推进的 DFS 迷宫生成器。
- 从全墙网格开始展示拆墙过程。
- 显示当前生成位置 `○`。
- 使用 ratatui 在 Terminal 中绘制 Unicode 线框迷宫。
- 正确显示入口 `S` 和出口 `E`。

验收标准：用户能看到墙体被逐步拆除，最终形成一张从入口连通到出口的迷宫。

### 里程碑 2：探索动画

- 实现 DFS 探索。
- 展示当前位置 `●`。
- 展示已探索路径 `·`。
- 到达出口后用双线字符展示最终路径。

验收标准：用户能看到探索者一步步找到出口。

### 里程碑 3：交互控制

- 支持暂停、继续、单步、调速、重新生成和退出。
- 增加底部状态栏。
- 增加命令行参数。

验收标准：用户可以在探索过程中控制节奏。

### 里程碑 4：扩展算法

- 增加 Prim、Kruskal、Aldous-Broder、Wilson 和 Recursive Division 迷宫生成。
- 增加 BFS、A*、Dijkstra、Dead-End Filling 和 Wall-Follower 探索。
- 可在运行前通过命令行参数选择生成算法和探索算法。
- 可在运行中使用数字键切换探索算法。
- 在状态栏显示当前生成算法和探索算法。

验收标准：不同算法的动画效果明显不同，且所有探索算法都能找到出口。

### 里程碑 5：体验打磨

- 增加 ASCII 降级模式。
- 处理 Terminal 尺寸不足。
- 默认根据 Terminal 最大可用区域自动计算迷宫逻辑尺寸。
- 处理运行时 Terminal resize，窗口变小时暂停并提示，窗口变大时保持当前迷宫并提示可重新生成。
- 增加随机种子。
- 优化边框、状态栏和提示文案。

验收标准：程序在不同 Terminal 尺寸下不出现明显渲染错乱。

## 20. 测试要点

功能测试：

- 生成器每一步最多只改变一对相邻单元格之间的共享墙。
- 未指定 `--width` 和 `--height` 时，迷宫逻辑尺寸按 Terminal 可用区域自动计算，且渲染结果能完整放入屏幕。
- 运行时窗口尺寸变化不会改变当前迷宫逻辑尺寸。
- 迷宫任意尺寸下都能从入口到达出口。
- 所有单元格都连通。
- 完美迷宫中任意两个单元格之间只有一条路径。
- 探索算法最终能找到出口。
- 最终路径从 `S` 连续连接到 `E`。

渲染测试：

- 生成阶段能从全墙网格逐步变化为完整迷宫。
- 当前生成位置 `○` 不会覆盖状态栏或破坏墙体连接。
- 外墙正确闭合。
- 入口和出口正确开口。
- Unicode 连接字符没有明显断裂。
- 状态栏不会覆盖迷宫。
- Terminal 尺寸不足时能给出清晰提示。
- 运行时缩小窗口导致当前迷宫放不下时，界面显示提示信息且不会渲染残缺迷宫。
- 运行时放大窗口时，当前迷宫保持不变，状态栏提示可以按 `N` 使用新尺寸重新生成。

交互测试：

- `Space` 能在 `Ready` 阶段开始探索，并能在动画阶段暂停和继续。
- `S` 能单步执行。
- `N` 能生成新迷宫。
- `R` 能重置当前迷宫的探索状态。
- `+` 和 `-` 能调整速度。
- `Q` 能退出程序。
